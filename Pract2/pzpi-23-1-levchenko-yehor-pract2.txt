3 ВСТУП
     Сучасні системи на основі великих мовних моделей дедалі частіше виходять за межі чат-інтерфейсів і стають повноцінними інструментами автоматизації. Вони редагують файли, виконують команди, взаємодіють із зовнішніми сервісами та координують роботу кількох паралельних агентів. Claude Code від компанії Anthropic є одним із найбільш показових прикладів такого підходу – це агентний інструмент командного рядка, який перетворює мовну модель на активного учасника процесу розробки програмного забезпечення.
     Ця доповідь присвячена дослідженню архітектури Claude Code. Вихідний код системи став доступним 31 березня 2026 року внаслідок витоку через файли вихідних карт у npm-пакеті дистрибуції. Репозиторій зберігся як архів для навчальних та дослідницьких цілей і став об'єктом аналізу архітектурних рішень, покладених в основу агентних систем нового покоління.
     Дослідження охоплює понад 1 900 файлів і більше 519 тисяч рядків коду. На основі цього матеріалу проаналізовано архітектурний стиль системи, її ключові компоненти, механізми взаємодії між ними, підходи до масштабованості та надійності. Окремо розглянуто питання, які виникають при проектуванні складних агентних систем, зокрема управління дозволами, оркестрація підагентів та гібридна хмарна модель розгортання.


4 ОПИС ВИКОНАНОЇ РОБОТИ
4.1 Загальні відомості про систему
     Claude Code – це агентний інструмент командного рядка, розроблений компанією Anthropic. Система надає розробникам можливість взаємодіяти з великою мовною моделлю Claude безпосередньо з термінального середовища, делегуючи їй реальні інженерні завдання: читання та редагування файлів, виконання команд оболонки, пошук по кодовій базі, роботу з системами контролю версій, а також координацію складніших, багатоетапних робочих процесів.
     Масштаб системи є значним навіть за мірками промислових проєктів. Кодова база налічує понад 1 900 файлів і більше 519 тисяч рядків коду, реалізує понад 53 окремих інструменти та підтримує понад 95 команд, доступних користувачеві. Для реалізації термінального інтерфейсу обрано середовище виконання Bun з мовою TypeScript у строгому режимі, декларативний UI побудований на React у зв'язці з бібліотекою Ink, яка адаптує реактивну парадигму до термінального рендерингу. Схеми конфігурації та вхідних даних визначаються через Zod v4, пошук по файлах забезпечує ripgrep, автентифікація реалізована через OAuth 2.0 та JWT, телеметрія збирається за допомогою OpenTelemetry.
     Ключова особливість системи полягає в тому, що вона не є звичайним чат-клієнтом із можливістю виклику функцій. Claude Code проектувалася як агентна система, де модель самостійно планує послідовність дій, виконує їх через інструменти, аналізує результати та коригує план на льоту. Саме цей агентний характер визначає більшість архітектурних рішень, прийнятих у системі.
4.2 Архітектурний стиль системи
     Аналіз вихідного коду дозволяє визначити архітектурний стиль Claude Code як модульну агентну архітектуру з подієво-орієнтованою координацією. Це поєднання трьох підходів, кожен із яких відповідає за окремий аспект системи.
     Модульність проявляється на рівні інструментів. Кожен інструмент є самодостатнім модулем, який оголошує схему вхідних даних, реалізує перевірку дозволів та містить логіку виконання. Агентний цикл взаємодіє з будь-яким інструментом через єдиний контракт, не знаючи нічого про його внутрішню реалізацію. Це рішення дозволяє додавати нові інструменти без зміни ядра системи та робить систему відкритою до розширення через плагіни, зовнішні MCP-сервери та навички, визначені користувачем.
     Агентна архітектура реалізована в центральному рушії – QueryEngine. Він підтримує ітеративний цикл, у якому модель генерує відповідь, система виявляє виклики інструментів, виконує їх та повертає результати у контекст наступного ходу. Цикл повторюється до тих пір, поки модель не припинить запитувати інструменти або не буде досягнуто одну з умов завершення. Цей підхід принципово відрізняється від моделі «запит-відповідь» і дозволяє системі вирішувати складні, багатокрокові задачі автономно.
     Подієво-орієнтована координація стосується взаємодії між агентами. Підагенти не викликаються напряму – вони запускаються, виконуються незалежно і сигналізують про завершення через структуровані XML-повідомлення, які потрапляють у чергу координатора як нові ходи розмови. Це рішення усуває жорстку зв'язаність між агентами та дозволяє реалізувати справжній паралелізм без складного механізму синхронізації.
     Вибір саме такого поєднання підходів не є випадковим. Він відображає природу задач, які система вирішує: агентна модель потрібна для автономного планування, модульність – для розширюваності, а подієва координація – для паралельної роботи без централізованого планувальника.
4.3 Основні компоненти системи
4.3.1 Термінальний інтерфейс
     Інтерфейс користувача Claude Code реалізований через React і бібліотеку Ink, яка переносить декларативну парадигму компонентів у термінальне середовище. Компонентний шар налічує понад 140 компонентів, організованих у функціональні групи.
     Центральним елементом введення є компонент PromptInput, який відповідає за парсинг команд, управління історією, відображення підказок та резервування горизонтального простору для компаньйона. Рендеринг транскрипту розмови забезпечується компонентом Messages у зв'язці з VirtualMessageList – останній реалізує віртуалізацію для продуктивної роботи з довгими розмовами. FullscreenLayout координує повноекранне відображення з прокручуваним транскриптом, модальними вікнами та закріпленим блоком введення.
     Система дозволів виокремлена в окрему групу компонентів. Для кожного типу інструменту існує власний компонент відображення запиту на підтвердження: для команд оболонки – BashPermissionRequest, для файлових операцій – FilePermissionDialog, для мережевих запитів – WebFetchPermissionRequest. Такий підхід дозволяє надавати користувачеві контекстно-залежний опис кожної операції, а не універсальне вікно підтвердження.
4.3.2 Ядро системи
     Точка входу системи, реалізована у файлі main.tsx на базі Commander.js, розпочинає роботу з паралельної ініціалізації: зчитування MDM-налаштувань, prefetch ключів автентифікації та попереднє підключення до API відбуваються одночасно ще до завантаження важких модулів. Це рішення суттєво скорочує час першого запуску.
     Центральним компонентом ядра є QueryEngine – рушій виконання запитів до мовної моделі. Файл налічує близько 46 тисяч рядків і реалізує: потоковий прийом відповідей від API, ітеративний цикл виклику інструментів, управління бюджетом токенів, логіку повторних спроб та відновлення після помилок. Поруч із ним розташований Tool.ts – базовий файл типів (~29 тисяч рядків), який визначає схеми вхідних даних, моделі дозволів та форми результатів для всіх інструментів системи.
     Система підтримує кілька режимів виконання. У стандартному режимі CLI виконує запити безпосередньо до Anthropic API або до сторонніх провайдерів – AWS Bedrock, Google Vertex, Azure Foundry. У серверному режимі CLI виступає клієнтом для зовнішнього direct-connect сервера. У режимі віддалених сесій (CCR) локальний процес керує UX та управляє потоком подій, тоді як саме виконання відбувається у хмарному середовищі.
4.3.3 Система інструментів
     Система інструментів є основним рівнем виконання дій у системі. Кожен із понад 53 інструментів реалізує єдиний контракт: оголошення схеми вхідних даних через Zod v4, перевірку дозволів, виконання логіки та повернення нормалізованого результату. Агентний цикл взаємодіє з будь-яким інструментом однаково, незалежно від того, чи йдеться про читання файлу, виконання bash-команди, пошук по кодовій базі чи запуск підагента.
     Серед ключових інструментів варто виділити AgentTool, який відповідає за запуск підагентів. Саме цей інструмент є точкою входу до багатоагентного виконання. MCPTool забезпечує виклик зовнішніх сервісів через протокол Model Context Protocol, LSPTool інтегрує мовні сервери для аналізу коду, а SkillTool виконує навички – багаторазово використовувані робочі процеси, визначені у вигляді markdown-файлів.
4.3.4 Система навичок
     Система навичок реалізує механізм, який дозволяє зберігати та повторно використовувати складні робочі процеси. Навичка визначається як markdown-файл з frontmatter-метаданими у директорії .claude/skills/ і може містити шаблон промпту, перелік дозволених інструментів, вказівки щодо моделі та стратегію виконання.
     Система підтримує кілька джерел навичок: вбудовані навички постачаються разом із системою, навички, визначені користувачем або проектом, завантажуються з файлової системи, а плагіни та MCP-сервери можуть постачати власні навички через той самий інтерфейс. Примітною особливістю є умовна активація: навичка може визначити шаблони шляхів файлів, при торканні яких вона стає доступною. Це дозволяє тримати навички поза контекстом до моменту, коли вони стають релевантними.
     Виконання навички може відбуватися двома способами. У стандартному режимі промпт навички ін'єктується у поточну розмову і обробляється в тому самому циклі. У форкованому режимі запускається окремий підагент із власним набором інструментів та обмежень, результат якого повертається до основного агента після завершення.
4.3.5 Координатор та багатоагентна оркестрація
     Координатор реалізує оркестрацію багатоагентних робочих процесів через prompt-driven підхід: спеціалізований системний промпт визначає поведінку координатора, а управління підагентами відбувається через стандартні інструменти. Координатор використовує AgentTool для паралельного запуску підагентів та TeamCreateTool для організації командної роботи.
     Важливо розуміти, що координатор – це не окремий процес-планувальник. Це той самий агентний цикл QueryEngine, але з іншим системним промптом та режимом поведінки. Надійність оркестрації залежить від явної реєстрації стану задач та черги повідомлень, а не від архітектурної ізоляції.
     Підагенти виконуються незалежно і сигналізують про завершення через структуроване XML-повідомлення, яке містить ідентифікатор задачі, статус, резюме та посилання на результат. Координатор отримує ці повідомлення як нові ходи розмови і на їх основі вирішує – продовжити роботу, скоригувати план або синтезувати фінальну відповідь для користувача. Такий конверт повідомлень є стабільним контрактом між підагентами та координатором.
     Міжагентна комунікація реалізована через два канали. Mailbox-система (SendMessageTool) підтримує направлені повідомлення між учасниками команди, включно з широкомовною розсилкою та структурованими управляючими повідомленнями. Для взаємодії з підагентами, запущеними в тому самому процесі, використовується внутрішня черга, яка доставляє повідомлення при наступному ході інструменту.
4.4 Взаємодія компонентів
4.4.1 Агентний цикл виконання
     Центральним потоком даних у системі є агентний цикл, реалізований у QueryEngine. Його розуміння є ключем до розуміння всієї архітектури системи.
     Цикл починається з отримання повідомлення користувача у методі submitMessage(). На цьому етапі формується контекст ходу: робочий каталог, модель, частини системного промпту та обгортка дозволів. Функція processUserInput() обробляє введення – парсить slash-команди, формує вкладення, застосовує зміни дозволів та перемикання моделі. Підготовлені повідомлення додаються до транскрипту і передаються у функцію query(), яка запускає ітеративний цикл queryLoop().
     У кожній ітерації циклу формується стиснений вигляд повідомлень, який передається у потоковий виклик API моделі. Відповідь, що надходить потоком, розбирається на блоки. Якщо серед них виявлено блоки типу tool_use, система диспетчеризує виклики інструментів. У потоковому режимі кожен блок негайно ставиться до черги StreamingToolExecutor для паралельного виконання. У непотоковому режимі всі зібрані блоки передаються у функцію runTools() для пакетного виконання. Результати інструментів нормалізуються і додаються до списку повідомлень наступного ходу, після чого цикл починається знову.
     Якщо відповідь моделі не містить блоків tool_use, цикл завершується з відповідним терміналним результатом. Серед умов завершення – нормальне завершення, перевищення максимальної кількості ходів, надмірний розмір контексту, помилка моделі або явне переривання користувачем. Зовнішній метод submitMessage() нормалізує потік подій, обліковує використання токенів та формує SDK-результат для передачі у UI.

4.4.2 Bridge-протокол взаємодії з IDE
     Bridge-система реалізує двоспрямований зв'язок між розширеннями IDE – Visual Studio Code та JetBrains – і CLI-процесом. Протокол розрізняє два логічних канали: канал даних для потоку повідомлень розмови та канал управління для операцій, які потребують підтвердження або зміни стану.
     Повідомлення каналу даних передають асистентські відповіді та події сесії від CLI до IDE. Канал управління використовується для операцій, які потребують зворотного зв'язку: зміна моделі, налаштування режиму дозволів та, найважливіше, запити на підтвердження виконання інструментів. Кожен такий запит містить унікальний ідентифікатор, що дозволяє системі зіставити відповідь IDE з конкретним очікуючим запитом.
     Процес запиту дозволу ілюструє цей механізм. Коли агентний цикл виявляє виклик інструменту, що потребує підтвердження, CLI формує повідомлення типу control_request з підтипом can_use_tool і надсилає його через Bridge. IDE відображає відповідний UI, користувач приймає рішення, і IDE повертає control_response з результатом – allow або deny. CLI, отримавши відповідь, розблоковує очікуючий виклик інструменту. Якщо відповідь не надходить протягом встановленого часу або запит скасовується, система повертає рішення deny із відповідним поясненням.
     Автентифікація в Bridge-системі розділена на два рівні. OAuth-токен використовується для управління середовищем та сесійними API-операціями. JWT ingress-токен призначений для каналів даних конкретної сесії і підлягає проактивному оновленню до закінчення терміну дії.

4.5 Масштабованість та надійність
4.5.1 Паралелізм на рівні агентів
     Claude Code вирішує задачу масштабованості не через горизонтальне масштабування інфраструктури, а через паралелізм на рівні агентів. Для кожної незалежної підзадачі координатор запускає окремий підагент через AgentTool. Підагенти виконуються паралельно, кожен зі своїм набором інструментів, обмежень та ізольованим станом.
     Ізоляція робочого середовища між підагентами може бути посилена через механізм Git Worktree: інструменти EnterWorktreeTool та ExitWorktreeTool переводять підагента у власне відгалуження репозиторію, що запобігає конфліктам при одночасному редагуванні файлів. TeamCreateTool дозволяє організувати постійнішу команду агентів із спільним сховищем задач і координацією через mailbox.
4.5.2 Відмовостійкість
     Відмовостійкість у системі реалізована на кількох рівнях, які взаємодоповнюють одне одного. На рівні API-клієнта централізована логіка повторних спроб (withRetry.ts) обробляє трансієнтні помилки, обмеження швидкості та ситуації перевантаження сервера. При систематичних збоях активується перехід на резервну модель.
     На рівні агентного циклу QueryEngine реалізує власні механізми відновлення. При перевищенні ліміту токенів контексту автоматично запускається компакція розмови. При помилках структурованого виводу виконується повторна спроба в межах встановленого ліміту. Bridge-система підтримує автоматичне перепідключення з keepalive-пінгами та обробкою постійних розривів.
     Варто також відзначити підхід fail-open, який свідомо застосовується у кількох місцях системи. Синхронізація налаштувань та проксі-конфігурація при помилках не блокують роботу системи – вони мовчки пропускаються, і сесія продовжується без них. Це рішення відображає пріоритет: краще дозволити роботу з неповними налаштуваннями, ніж заблокувати весь робочий процес.
4.5.3 Кешування та оптимізація контексту
     Одним із найцікавіших технічних рішень у системі є багаторівнева система управління контекстом. Розмова з мовною моделлю обмежена вікном контексту, і в довгих сесіях це стає серйозним обмеженням. Claude Code вирішує цю проблему через кілька взаємодоповнювальних механізмів.
     Автоматична компакція (autoCompact) відстежує кількість використаних токенів і при наближенні до порогу запускає стиснення розмови. Circuit breaker запобігає зациклюванню у ситуаціях, коли компакція сама по собі вичерпує контекст. MicroCompact стискає результати окремих інструментів і підтримує кешування промптів, зменшуючи вартість повторних запитів. Сервіс extractMemories у фоновому режимі витягує важливі факти з транскрипту і зберігає їх у файли постійної пам'яті, які підключаються до контексту майбутніх сесій.
     Оптимізація часу запуску реалізована через паралельний prefetch та відкладене завантаження. Важкі модулі – OpenTelemetry, gRPC, аналітика – завантажуються лише при першому зверненні. Feature flags через механізм Bun дозволяють повністю вилучити неактивний код під час збірки, що зменшує розмір бінарного файлу та прискорює запуск.
4.5.4 Реплікація даних та хмарна інфраструктура
     Аналіз вихідного коду src/services/ виявив, що явна реплікація даних у системі відсутня. Claude Code не реалізує управління replica-set, quorum або leader election. Це закономірно: система є клієнтським інструментом, а не розподіленою базою даних. Довговічність та реплікація даних делегуються хмарним бекенд-сервісам Anthropic, які споживаються через API.
     Натомість система реалізує власні механізми узгодженості та довговічності. Командна синхронізація пам'яті використовує оптимістичну конкурентність із ETag та If-Match заголовками і дельта-завантаженнями – лише змінені записи завантажуються на сервер. Журнал транскриптів сесій підтримує послідовний append-лог з відновленням після конфліктів. Локальні файли постійної пам'яті та витягнуті спогади зберігаються у файловій системі.
     Модель розгортання системи є гібридною. Локальна частина включає CLI-оркестрацію, виконання інструментів, термінальний UI та локальний стан. Хмарні сервіси Anthropic надають API виведення моделі, бекенд сесій із WebSocket-підписками, канали управління, сховище для синхронізації пам'яті та налаштувань, а також журнал транскриптів. Залежно від режиму, виконання може залишатися переважно локальним або переміщуватися у віддалену сесію CCR, де локальний процес виступає лише клієнтом керуючої площини.
4.6 Критичний аналіз
     Архітектура Claude Code містить низку рішень, які, попри зовнішню логічність, несуть підтверджені ризики. Цей розділ базується на результатах дослідження вихідного коду і розрізняє підтверджені проблеми від підозрілих місць, що потребують подальшої верифікації.
4.6.1 QueryEngine як God Object
     Найбільшою структурною проблемою системи є QueryEngine. Файл обсягом близько 46 тисяч рядків реалізує в одному методі submitMessage() одночасно: збирання системного промпту, ін'єкцію пам'яті, обгортку дозволів, препроцесинг slash-команд, управління транскриптом, оркестрацію агентного циклу, нормалізацію потокових подій, обробку компакції, контроль бюджету токенів та повторні спроби структурованого виводу. Це класичний God Object – компонент, зміна будь-якого аспекту якого потенційно зачіпає всі інші.
     Дослідження коду підтвердило також пряму залежність між оркестратором і конкретним інструментом: QueryEngine безпосередньо перевіряє SYNTHETIC_OUTPUT_TOOL_NAME та керує логікою структурованого виводу. Це порушує принцип, на якому побудована решта системи, – єдиний контракт інструменту. Поряд із цим Tool.ts, базовий файл типів обсягом близько 29 тисяч рядків, є точкою залежності для всіх інструментів системи: зміна спільного типу в цьому файлі потенційно зачіпає понад 97 споживачів одночасно. ToolUseContext, що збирається у цьому ж файлі, поступово перетворюється на неявний service locator, агрегуючи взаємно незалежні можливості, які більшість інструментів не використовує.
4.6.2 Крихкість координатора
     Багатоагентна координація є однією з найцікавіших можливостей системи, але саме тут сконцентровані найсерйозніші підтверджені проблеми. Ключова з них стосується черги task-notification: повідомлення про завершення підагентів існують виключно в оперативній пам'яті процесу. Якщо координатор завершує роботу до того, як черга спустошена, ці повідомлення втрачаються назавжди, і координатор не дізнається про завершення своїх підагентів.
     Пов'язана проблема стосується поведінки при вичерпанні ліміту ходів. Коли підагент досягає max_turns, цикл завершується з позначкою max_turns_reached, але ця інформація може не дійти до координатора у вигляді явного сигналу невдачі. Координатор здатний отримати повідомлення, яке виглядає як нормальне завершення задачі, не усвідомлюючи, що робота не була виконана до кінця. Нарешті, якщо координатор сам вичерпує контекст або ліміт ходів під час очікування результатів, підагенти продовжують роботу і споживати ресурси без будь-якого нагляду – механізму автоматичного припинення підагентів при завершенні координатора не передбачено.
4.6.3 Проблеми синхронізації командної пам'яті
     Синхронізація командної пам'яті (teamMemorySync) реалізує складну логіку узгодженості, в якій дослідження виявило кілька підтверджених проблем. При конфлікті запису (412 від сервера) система застосовує стратегію local-wins: клієнт перераховує дельту від поточного локального стану і перезаписує ключ на сервері. Якщо два учасники команди одночасно змінюють один і той самий запис пам'яті, зміни одного з них будуть мовчки перезаписані. Коментарі у вихідному коді явно фіксують відсутність механізму злиття – це не упущення, а свідомий вибір, але його наслідки для командних робочих процесів є суттєвими.
     Другою підтвердженою проблемою є відсутність атомарності пакетних завантажень. Дельта розбивається на пакети і завантажується послідовно; успішно завершені пакети оновлюють стан синхронізації до того, як наступний пакет завершиться. У разі помилки в середині процесу сервер опиняється у частково оновленому стані, хоча операція в цілому позначається як невдала. Нарешті, debounce-механізм у watcher.ts та best-effort flush при завершенні процесу створюють вікно, в якому нещодавні локальні зміни можуть бути мовчки втрачені при примусовому завершенні процесу.
4.6.4 Компакція: механізм відновлення, що може відмовити
     Автоматична компакція є критично важливим механізмом для довгих сесій, але її реалізація містить кілька вразливостей. Після трьох послідовних невдач autoCompact вмикає circuit breaker і припиняє подальші спроби компакції на весь час сесії. Якщо невдачі були тимчасовими і сесія продовжується, вона наближається до лімітів контексту без жодного автоматичного захисту. Система опиняється у стані, де механізм відновлення від перевантаження контексту сам відключився через перевантаження контексту.
     При компакції, яка сама перевищує контекстне вікно, система застосовує стратегію усічення: найстаріші групи повідомлень відкидаються і виконується повторна спроба. Якщо й повторна спроба не вдається, компакція завершується з помилкою для користувача. Це означає незворотну втрату частини контексту сесії. Водночас варто відзначити, що одна потенційно небезпечна проблема підтвердження не отримала: коду, що гарантує збереження пар tool_use/tool_result при компакції, існує і явно перевіряє цей інваріант – це захищає від цілого класу API-помилок.
4.6.5 SendMessageTool: сім семантик в одному інструменті
     SendMessageTool обслуговує сім різних транспортних шляхів в одній функції call(): bridge-маршрутизацію, UDS-маршрутизацію, in-process чергування для підагентів, відновлення зупинених агентів, mailbox для прямих повідомлень, широкомовну розсилку командою та структуровані управляючі повідомлення. Ці шляхи не ізольовані за окремими обробниками – вони чергуються у спільному потоці керування. Зміна поведінки одного шляху потенційно впливає на всі інші.
     Широкомовна розсилка (broadcast) реалізована як послідовний запис до mailbox кожного учасника команди без транзакційного відкату. Якщо запис до одного учасника завершується помилкою, частина команди отримає повідомлення, а частина – ні, і стан розповсюдження команди стає розбіжним. Система не надає спостережливості щодо того, який саме маршрут спрацював при конкретній відправці – це ускладнює діагностику у виробничому середовищі.
4.6.6 Гібридна модель розгортання та режимна амбівалентність
     Система підтримує кілька режимів виконання – локальний, CCR, bridge, remote – але вибір режиму не керується єдиним явним перемикачем. Натомість кожна підсистема незалежно перевіряє власний набір умов і прапорців. Це означає, що часткова активація режиму є можливою: наприклад, проксі-конфігурація для remote-режиму може бути пропущена через fail-open при ініціалізації, тоді як інші підсистеми вже перейшли у remote-режим. Система продовжить роботу, але у деградованому стані, який може бути важко діагностувати.
4.6.7 JWT без перевірки підпису
     Bridge-система декодує JWT-токени без криптографічної перевірки підпису. Дослідження підтвердило, що декодовані дані використовуються виключно для планування оновлення токена, а не для прийняття рішень щодо доступу – це суттєво обмежує практичну поверхню атаки. Проте маніпуляція значенням exp у токені може призвести до патернів відмови: надто далеке майбутнє значення затримає оновлення токена, надто близьке – викличе агресивні повторні спроби або передчасне завершення сесії. Це не критична вразливість, але є поведінкою, яка залежить від цілісності даних, що не перевіряється.
4.7 Приклади програмної реалізації
     Приклади програмної реалізації Claude Code, наведені у додатку В, виконані у мово-нейтральному псевдокоді та ілюструють ключові архітектурні принципи системи. Вони не є копіями вихідного коду.
     Також, у додатку В наведено перелік запитів до агенту штучного інтелекту, що здійснював збір та аналіз кодової бази, та формував на їх основі звіти, які можна знайти у GitHub репозиторії.

5 ВИСНОВКИ
     У результаті виконання практичного заняття:
* досліджено архітектуру програмної системи Claude Code;
* визначено її архітектурний стиль;
* проаналізовано взаємодію компонентів;
* оцінено переваги та обмеження архітектурних рішень;
* сформовано практичні навички архітектурного аналізу ПЗ.
     Аналіз архітектури Claude Code засвідчує фундаментальний зсув у розробці програмного забезпечення: оркестрація складних процесів дедалі більше делегується самій мовній моделі через семантичні XML-контракти та гнучкі воркфлоу. Система доводить, що модульна агентна архітектура з подієво-орієнтованою координацією вже є життєздатною та готовою до використання як потужний інструмент для локальної розробки.
     Водночас, інноваційність концепцій не забезпечила бездоганної організації вихідного коду. Наявність виражених антипатернів свідчить про накопичений технічний борг та проблеми з розділенням відповідальності компонентів. Крім того, архітектура містить низку жорстких компромісів: відмова від персистентних черг на користь in-memory маршрутизації, агресивне відкидання контексту та оптимістичне блокування без механізмів злиття. Розробники свідомо пожертвували надійністю розподіленого стану заради максимальної швидкодії CLI-клієнта та економії токенів.
     Таким чином, на даному етапі Claude Code є високоефективним рішенням переважно для соло-інженерів. Для безпечного застосування у великих командах механізми управління станом та синхронізації потребують значного доопрацювання. Головний підсумок дослідження полягає в тому, що створення надійних агентних систем – це виклик не стільки у сфері штучного інтелекту, скільки у класичній програмній інженерії: організації чистого коду та побудові стійкої інфраструктури навколо непередбачуваної мовної моделі.

      1. FUNCTION queryLoop(initialMessages, modelClient, toolRegistry, loopOptions):
      2.   messages = initialMessages
      3.   turnCount = 0
      4. 
      5.   WHILE true:
      6.     turnCount = turnCount + 1
      7. 
      8.     modelResponse = modelClient.generate(messages)
      9.     IF modelResponse.error:
      10.       RETURN TerminalResult(status="error", reason="model_error")
      11. 
      12.     assistantMessage = modelResponse.assistantMessage
      13.     YIELD assistantMessage
      14. 
      15.     toolUseBlocks = extractBlocks(assistantMessage, type="tool_use")
      16.     IF toolUseBlocks.isEmpty:
      17.       RETURN TerminalResult(status="success", finalMessage=assistantMessage)
      18. 
      19.     toolResults = []
      20.     FOR EACH toolUseBlock IN toolUseBlocks:
      21.       tool = toolRegistry.find(toolUseBlock.name)
      22.       IF tool IS NULL:
      23.         toolResults.add(makeToolError(toolUseBlock.id, "Unknown tool"))
      24.         CONTINUE
      25. 
      26.       permissionDecision = tool.checkPermission(toolUseBlock.input, loopOptions.permissionContext)
      27.       IF permissionDecision.behavior = "deny":
      28.         toolResults.add(makeToolError(toolUseBlock.id, permissionDecision.reason))
      29.         CONTINUE
      30. 
      31.       executionResult = tool.execute(toolUseBlock.input, loopOptions.runtimeContext)
      32.       toolResults.add(makeToolResult(toolUseBlock.id, executionResult))
      33. 
      34.     FOR EACH toolResultMessage IN toolResults:
      35.       YIELD toolResultMessage
      36. 
      37.     messages = messages + [assistantMessage] + toolResults
      38. 
      39.     IF turnCount >= loopOptions.maxTurns:
      40.       RETURN TerminalResult(status="error", reason="max_turns")

      1. FUNCTION runCoordinator(userGoal, workerPool, notificationQueue):
      2.   taskList = planTasks(userGoal)
      3.   taskState = Map()  // taskId -> {status, summary, outputRef}
      4. 
      5.   FOR EACH task IN taskList:
      6.     taskId = createTaskRecord(task)
      7.     taskState[taskId] = {status: "queued"}
      8. 
      9.   // Fan-out: spawn workers in parallel
      10.   FOR EACH taskId IN taskState.keys PARALLEL:
      11.     taskSpec = loadTaskSpec(taskId)
      12.     workerHandle = workerPool.spawn(taskSpec)
      13.     taskState[taskId].status = "running"
      14.     taskState[taskId].workerHandle = workerHandle
      15. 
      16.   completedCount = 0
      17.   WHILE completedCount < taskList.length:
      18.     taskNotification = notificationQueue.waitFor("task-notification")
      19.     taskId = taskNotification.taskId
      20. 
      21.     IF taskNotification.status = "completed":
      22.       taskState[taskId].status = "completed"
      23.       taskState[taskId].summary = taskNotification.summary
      24.       taskState[taskId].outputRef = taskNotification.outputFile
      25.       completedCount = completedCount + 1
      26.     ELSE IF taskNotification.status = "failed":
      27.       taskState[taskId].status = "failed"
      28.       taskState[taskId].summary = taskNotification.summary
      29.       completedCount = completedCount + 1
      30.     ELSE:
      31.       // optional progress update
      32.       taskState[taskId].status = taskNotification.status
      33. 
      34.   synthesisInput = collectWorkerOutputs(taskState)
      35.   finalAnswer = synthesizeFinalResponse(userGoal, synthesisInput)
      36.   RETURN finalAnswer
